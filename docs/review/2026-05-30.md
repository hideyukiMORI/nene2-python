## コードベース評価レポート（AI 所見）

### 今日見えてきたこと

**「検証していない約束」がバグを生む**  
「sqlite/mysql/pgsql 対応」と謳い、設定・コードも書いてあったが、実 DB でテストしたことが一度もなかった。捨てコンテナで実測した途端に `save()` が常に `1` を返すバグが出た。`ignoreDeadLinks: true` で docs が 4 回連続デプロイ失敗していたのも同じ構造で、「設定が存在する」と「実際に機能している」は別の問題。外部状態は推測せず実測する、というのがこのセッションの通奏低音だった。

**インフラが先に整うと後の作業が安全になる**  
dead-link 検出・サイドバー網羅チェック・PR ゲート化（#759/#760）を整えてから 6 言語化（#765/#766）に入ったので、148 ファイルのマージ後にビルドが壊れなかった。逆順だったら 4 言語分の壊れたページが公開されていた。

**MCP パリティの教訓**  
「両サーフェスで成り立つ規則は UseCase に置け」というドキュメントの原則（#751）を、リファレンスアプリ自身が破っていた（#753）。docs と実装の乖離は気づきにくい。パリティテスト（`tests/example/test_http_mcp_parity.py`）が「ドキュメントの主張を CI で検証する」役割を担うようになったのは一つの進歩。

**#545（NeNe 流 FT ガバナンス）の決着スタイルについて**  
提案文書をそのままマージせず、唯一まだ無かった friction/decision タクソノミーだけを既存 docs に吸収して close した。「借り物の形式を丸ごと取り込まず、必要な 1 点だけ取る」判断は、フレームワークの「薄さ」の哲学と整合していた。
