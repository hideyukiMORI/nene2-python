## 概要

`html` モジュールは HTML 特殊文字のエスケープ/アンエスケープを提供する。XSS 対策の基本だが「`html.escape` だけで全コンテキストが安全になる」という誤解が多い。本 FT では `quote=True` による属性コンテキストの安全性と、**コンテキスト依存性**（HTML テキスト/属性は守れるが JS/URL/CSS は別）を整理した。

| API | ユースケース |
|---|---|
| `html.escape(s, quote=True)` | `& < > " '` をエンティティ化（属性コンテキスト安全） |
| `html.unescape(s)` | 名前付き/数値エンティティを文字に復元 |

## 摩擦点

### F-1: `quote=True`（デフォルト）でないと属性コンテキストで XSS が通る

**観察**: `html.escape(s, quote=False)` は `& < >` のみエスケープし、`"` と `'` を残す。これを HTML 属性値（`<a title="...">`）に埋めると、`" onmouseover='alert(1)'` のようなペイロードで**属性を脱出**して XSS が成立する。

**対処**: `html.escape(text, quote=True)`（デフォルト）を使い `"` → `&quot;`、`'` → `&#x27;` もエスケープ。属性コンテキストでも安全。テストで両クォートのエスケープを確認。

### F-2: `&` が最初にエスケープされる（二重エスケープの順序）

**観察**: `html.escape` は `&` → `&amp;` を**最初に**処理する。順序を誤って自前実装すると `<` → `&lt;` の後に `&` → `&amp;` をやって `&amp;lt;` と二重エスケープする事故が起きる。標準 `html.escape` は正しい順序を保証。

**対処**: 自前置換ではなく `html.escape` をそのまま使う。`"a & b < c"` → `"a &amp; b &lt; c"`。

### F-3: `html.escape` はコンテキスト依存 — JS/URL/CSS には不十分

**観察**: `html.escape` が守るのは **HTML テキスト/属性コンテキスト**のみ。`<script>var x="...";</script>` の中（JS 文字列）、`href="javascript:..."`（URL）、`style="..."`（CSS）には別のエスケープ/検証が必要。`html.escape` を万能と誤解すると JS コンテキスト XSS が残る。

**対処**: 本 FT は HTML テキスト/属性に限定。JS は JSON シリアライズ、URL は `urllib.parse.quote`（FT228）、危険スキームの排除などコンテキスト別の対策を併用する旨をドキュメント化。
