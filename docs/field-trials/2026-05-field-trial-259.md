## 概要

`weakref` は**オブジェクトの生存を妨げない参照**を提供する。`WeakValueDictionary` は値への弱参照を持つキャッシュで、他に強参照がなくなったオブジェクトを自動退避する（メモリリーク防止）。HTTP API でラップし自動退避の挙動を検証した。

| API | ユースケース |
|---|---|
| `weakref.WeakValueDictionary` | 値が GC されると自動的にエントリ消滅 |
| `weakref.ref(obj)` | 弱参照（対象が生きていれば取得、死んでいれば None） |

## 摩擦点

### F-1: int/str/tuple は弱参照できない — 専用クラスが要る

**観察**: `weakref.ref(5)` や組み込みの int/str/tuple は弱参照を**サポートしない**（`TypeError: cannot create weak reference`）。`__slots__` を使うクラスは `__weakref__` スロットを明示しないと弱参照できない。

**対処**: `_Box` クラスに `__slots__ = ("value", "__weakref__")` を定義。弱参照可能にする。

### F-2: 強参照が残るとキャッシュから退避されない（リーク注意）

**観察**: `WeakValueDictionary` はキャッシュ自体が強参照を持たないので GC を妨げないが、**他のどこかで強参照を保持**していると退避されない。ループ変数（`box`）が最後のオブジェクトを掴み続けるだけでも 1 個残る（実装中に survivors が +1 になる罠を発見）。

**対処**: 不要になった強参照を `del` で明示解放してから `gc.collect()`。診断で keep 個だけが残り、残りは auto_evicted を確認。

### F-3: GC タイミングと決定性

**観察**: 自動退避は GC に依存し、CPython では参照カウントが 0 になった時点で即座に解放されるが、循環参照は `gc.collect()` を要する。テストの決定性のため明示的に `gc.collect()` を呼ぶ。

**対処**: テストでは `gc.collect()` で決定化。本番のキャッシュ用途では退避タイミングに依存しない設計にする。
