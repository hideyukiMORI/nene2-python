## 概要

`mimetypes.guess_type` は**ファイル名の拡張子**から MIME を推測する。アップロード検証で重要なのは「**クライアント申告の Content-Type は信頼できない**（偽装可能）」こと。拡張子の許可リスト + 申告型一致で検証する設計を診断した。完全な防御にはさらにマジックバイト検査と「アップロードファイルを実行しない」運用が必要。

| API | ユースケース |
|---|---|
| `mimetypes.guess_type(filename)` | 拡張子からの MIME 推測（None もあり得る） |
| 拡張子許可リスト | 許可する拡張子→期待 MIME |

## 摩擦点

### F-1: クライアント申告 Content-Type を信頼しない

**観察**: HTTP の `Content-Type` ヘッダーや multipart の content-type は**クライアントが自由に詐称できる**。`image/png` と申告しつつ実体が `.php` ということが容易にできる。これを信頼すると危険ファイルのアップロードを許す。

**対処**: **拡張子の許可リスト**（`.png/.jpg/.jpeg/.pdf/.txt`）で受理を制限し、申告 Content-Type が拡張子由来の期待型と**一致**することも要求。`img.png` + `text/html` 申告（型偽装）を 422。

### F-2: 二重拡張子は最終拡張子で評価する

**観察**: `evil.png.php` のような二重拡張子は、サーバー設定次第で `.php` として実行される（古い Apache）。逆に `evil.php.png` は最終拡張子 `.png`。

**対処**: `PurePosixPath(filename).suffix`（最終拡張子のみ）で判定。`evil.png.php` は `.php` で 422、危険拡張子（.php/.exe）・拡張子なしも 422。

### F-3【限界】拡張子・Content-Type は「ヒント」— 内容検査が必要

**観察**: `evil.php.png`（最終 .png）は本 FT の検証を通る（200）。これは**正しく設定されたサーバーでは PNG として扱われ安全**だが、ファイルの**中身が PHP ペイロード**である可能性は残る。拡張子も Content-Type も内容を保証しない。

**対処（限界の明示）**: 完全な防御には ① **マジックバイト検査**（実際のファイル先頭を確認、`python-magic` 等）、② **アップロードを実行しない**（no-exec ディレクトリ・別ドメイン配信）、③ 画像は再エンコードして無害化、を併用する。本 FT は拡張子 + Content-Type の一次検証に焦点を当て、この限界を明記する。
