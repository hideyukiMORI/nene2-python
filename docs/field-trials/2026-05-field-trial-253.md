## 概要

`copy.copy`（浅い）と `copy.deepcopy`（深い）の違いは、ネストした可変オブジェクトの**共有/独立**を左右する重要な概念。HTTP API でラップし、2 次元配列の内側要素を変更したときの原本への波及を検証した。

| API | 性質 |
|---|---|
| `copy.copy(x)` | 浅い: 外側のみ複製、内側は共有 |
| `copy.deepcopy(x)` | 深い: 内側まで独立（循環は memo で処理） |

## 摩擦点

### F-1: 浅いコピーは内側の可変オブジェクトを共有する

**観察**: `copy.copy(matrix)` は外側 list を新規作成するが、**内側 list は原本と同一参照**。`shallow[r][c] = v` は原本の `matrix[r][c]` も変える。設定オブジェクトやデフォルト引数を浅くコピーして使い回すと、意図しない共有状態のバグになる。

**対処**: ネストした構造を独立させたいなら `deepcopy`。診断で `shallow_affects_original=True` / `deep_affects_original=False` を確認。トップレベルへの追加（行追加）は浅いコピーでも独立（`shallow_top_is_independent=True`）。

### F-2: deepcopy は循環参照を memo で安全に処理

**観察**: `deepcopy` は内部の memo 辞書で**既に複製したオブジェクトを記録**し、循環参照（`a = []; a.append(a)`）でも無限ループせず複製できる。JSON には循環がないが、ドメインオブジェクトでは起こり得る。

**対処**: deepcopy に委譲（自前再帰コピーは循環で破綻しやすい）。

### F-3: deepcopy のコスト — サイズ上限で DoS を防ぐ

**観察**: `deepcopy` は全要素を再帰的に複製するため、巨大構造で CPU・メモリを消費する。

**対処**: 行数・列数を上限化（100×100）。本番では入力サイズに応じた上限を設定。
