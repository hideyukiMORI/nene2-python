## 概要

`email.message.EmailMessage` はメールを構築する。HTTP API でラップする際の脅威は **ヘッダーインジェクション** — 件名や宛先に CRLF を仕込むと `Bcc: attacker@evil.com` のような**追加ヘッダー**を注入し、第三者へのスパム送信・本文改ざんにつながる。ペンテスト回（256 % 4 = 0）として CRLF 注入を攻撃した。

| API | ユースケース |
|---|---|
| `EmailMessage()` | メール構築 |
| `message["Subject"] = ...` | ヘッダー設定 |
| `message.set_content(body)` | 本文設定 |

## 摩擦点

### F-1: ヘッダー値の CRLF を拒否する（ヘッダーインジェクション）

**観察**: 件名 `"Hello\r\nBcc: attacker@evil.com"` をそのままヘッダーに入れると、`Bcc` ヘッダーが注入され攻撃者に密かに送信される。本文への改行注入でヘッダーと本文の境界を破ることもできる。

**対処**: `_validate_header_value` で CR/LF と制御文字（ord < 32）を拒否。件名・宛先のいずれの CRLF/LF/CR/null/tab も 422（本文の改行は許可 — 本文は改行が正当）。

### F-2: 多層防御 — Python の email ポリシーも CRLF を拒否する

**観察**: 参考として素の `EmailMessage["Subject"] = "Hi\r\nBcc: evil"` を試すと、Python 自身が `ValueError: Header values may not contain linefeed or carriage return characters` を送出する。**標準ライブラリも CRLF ヘッダー注入を防御**している（多層防御）。

**対処**: アプリ側の事前検証 + Python の email ポリシーの二重で防御。アプリ側で弾くことで 422 の明確なエラーを返せる。

### F-3【限界】簡易メールアドレス検証の permissive 性

**観察**: 本 FT の `_EMAIL_RE`（`^[^@\s]+@[^@\s]+\.[^@\s]+$`）は実用的だが厳密でなく、`<script>@x.com` のような奇妙なローカル部を受理する（200）。CRLF を含まないためヘッダーインジェクションにはならないが、厳密な検証ではない。

**対処（限界の明示）**: 厳密な検証が必要なら `email.headerregistry.Address` のパースや `email-validator` ライブラリを使う。メールアドレスを表示・再利用する際は別途エスケープ（FT229）。RFC 5322 完全準拠の正規表現は非現実的なため、本 FT は CRLF 遮断（注入対策）に焦点を当てる。
