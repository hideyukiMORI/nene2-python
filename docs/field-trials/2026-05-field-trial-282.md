# FT282: platform — システム情報を露出しない

**日付**: 2026-05-29

**セキュリティ診断**: 🔒 あり（282 % 3 = 0）

## 概要

`platform` はシステム情報（OS・バージョン・hostname・アーキテクチャ）を取得する。これらを **API レスポンスに含めると情報漏洩**になり、攻撃者のフィンガープリント・CVE 標的化を助ける（例: 正確な Python/OS バージョンから既知脆弱性を狙う）。診断回（282 % 3 = 0）として、詳細は内部ログのみ・公開は粗い OS family のみ、という原則を検証した。

| 情報 | 公開 |
|---|---|
| `platform.node()`（hostname） | **非公開** |
| `platform.python_version()` | **非公開** |
| `platform.release()`/`platform()` | **非公開** |
| 粗い OS family（posix/windows） | 公開可（非機微） |

## 摩擦点

### F-1: 詳細バージョン・hostname を API レスポンスに含めない

**観察**: ヘルスチェックや info エンドポイントで `platform.python_version()`・`platform.platform()`・`platform.node()` を返すと、攻撃者がスタック・バージョンを特定し既知 CVE を狙える。hostname は内部ネットワーク構造のヒントにもなる。

**対処**: レスポンスは `{healthy, os_family}` のみ（os_family は posix/windows/other の粗い分類）。診断で hostname・python version・OS release・platform()・machine がいずれも**レスポンスに含まれない**ことを確認。

### F-2: 詳細は内部ログにのみ記録する

**観察**: デバッグには詳細情報が必要だが、それは**サーバー内部のログ**で十分。クライアントに返す必要はない。

**対処**: `platform.system/release/python_version/node` を `logger.info`（内部）で記録し、レスポンスには出さない。観測性とセキュリティを両立。

### F-3: `Server` ヘッダー等の他の漏洩経路

**観察**: アプリのレスポンス本文だけでなく、`Server` ヘッダー・エラースタックトレース（FT エラーハンドリング）・依存バージョンも漏洩経路。

**対処**: 本 FT は本文に focus。スタックトレース非公開（`APP_DEBUG=false`）・`Server` ヘッダー抑制も併用する（CLAUDE.md のエラーハンドリングポリシーと整合）。
