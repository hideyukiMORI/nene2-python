## 概要

`fractions.Fraction` は**厳密な有理数**を扱う（自動既約化）。HTTP API でラップし合計と有理数近似（`limit_denominator`）を検証した。`decimal`（FT250）が十進固定精度なのに対し、`Fraction` は `1/3` のような循環小数も誤差なく扱える。

| API | ユースケース |
|---|---|
| `Fraction("1/3")` / `Fraction("0.25")` | 文字列から厳密生成 |
| 算術演算 | 自動既約化（`2/4`→`1/2`） |
| `.limit_denominator(n)` | 分母 n 以下の最良近似 |

## 摩擦点

### F-1: 巨大分母の制限は用途で出し分ける

**観察**: 合計では入力の分母が暴走（巨大化）しないよう上限が要る。一方 `approximate` は高精度小数（例: `3.141592653589793` = 分母 10^15）を**意図的に受けて縮約**するのが目的。最初、合計用の分母上限を近似にも適用したため、pi の近似が 422 になった（実装中に発見）。

**対処**: `_to_fraction(value, check_denominator=...)` で分母チェックを切替え。合計は `True`（巨大分母拒否）、近似は `False`（縮約前提で許容）。

### F-2: 文字列入力で精度を保つ（float 経由は誤差）

**観察**: `Fraction(0.1)`（float から）は `3602879701896397/36028797018963968` になる。`Fraction("0.1")` は `1/10` で厳密。

**対処**: 入力は `str` 受信。`"1/3"+"1/6"` → `"1/2"` を厳密に確認。

### F-3: ゼロ除算・不正入力

**観察**: `Fraction("1/0")` は `ZeroDivisionError`、`Fraction("abc")` は `ValueError`。

**対処**: 両方捕捉して 422。文字列長も制限。
