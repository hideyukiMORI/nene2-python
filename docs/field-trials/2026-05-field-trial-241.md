## 概要

`bisect` はソート済み列に対する二分探索（O(log n)）と挿入を提供する。HTTP API でラップし挿入位置・順位・範囲カウントを検証した。最大の落とし穴は **「入力がソート済み」という前提条件**で、未ソート列に使うと黙って誤った結果を返す。

| API | ユースケース |
|---|---|
| `bisect.bisect_left/right` | 挿入位置（重複の前/後） |
| `bisect.insort` | ソート順を保った挿入 |
| 範囲カウント | `bisect_right(hi) - bisect_left(lo)` |

## 摩擦点

### F-1: `bisect` はソート済みが前提 — 未ソートだと黙って誤る

**観察**: `bisect` は入力がソート済みであることを**前提とし検証しない**。未ソート列に `bisect_left` を使うと、例外を出さず**誤った位置**を返す。バグが silent に潜むため危険。

**対処**: `_require_sorted()` で昇順を明示検証し、未ソートは 422。`[3,1,2]` が拒否されることを確認。

### F-2: `bisect_left` と `bisect_right` の重複扱いの違い

**観察**: 重複値があるとき `bisect_left` は重複の**前**、`bisect_right` は**後**の位置を返す。混同すると順位計算がずれる。出現回数は `right - left` で求まる。

**対処**: 両方を返し違いを明示。`[1,2,2,2,3]` の `2` で left=1 / right=4 / occurrences=3 を確認。

### F-3: 範囲カウントは `bisect_right(hi) - bisect_left(lo)`

**観察**: `[low, high]` 両端含む件数は `bisect_right(high) - bisect_left(low)` で O(log n)。線形走査より高速だが、`low <= high` の検証は別途必要。

**対処**: `low > high` を 422。点数上限も設定。
