## 概要

glob パターンでファイルを検索する際、ユーザー制御のパターンが `../` や絶対パスを含むと**作業ディレクトリ外**を列挙できる（情報漏洩）。HTTP API でラップし、パターン検証 + `resolve()` + `is_relative_to()` による封じ込めを診断した。CLAUDE.md のパス操作ポリシーに従い、`glob.glob` ではなく **`Path.glob`/`rglob`** を使う（pathlib 推奨・`root_dir` 結合の罠を避ける）。

| API | ユースケース |
|---|---|
| `Path.glob(pattern)` | base 相対の glob 検索（`**` 対応） |
| `Path.resolve()` + `is_relative_to()` | 結果の封じ込め |

## 摩擦点

### F-1: ユーザーパターンの `..`・絶対パスを拒否する

**観察**: `glob("../*")` や `glob("/etc/*")` は作業ディレクトリ外を列挙する。`sub/../../*` のような埋め込み親参照も脱出経路。

**対処**: `_validate_pattern` で先頭 `/`（絶対）・`..`（親参照）・`\`（Windows 区切り）を拒否。診断で `../*`・`/etc/*`・`sub/../../*`・`**/../*` がすべて 422。

### F-2: `glob.glob(root_dir=...)` より `Path.glob` が安全

**観察**: `glob.glob(pattern, root_dir=base)` は base を基準にするが、pattern 内のトラバーサルとの結合で挙動が分かりにくい。ruff も `PTH207` で `Path.glob`/`rglob` を推奨。

**対処**: `base.glob(pattern)` を使い、各結果を `resolve()` して `is_relative_to(base)` で二重確認。pathlib なので CLAUDE.md のパス操作ポリシーとも整合。

### F-3: `**`（再帰 glob）のコスト

**観察**: `**/*` は再帰的に全階層を走査するため、深い/広いツリーでコストが増す。

**対処**: ワークスペースは小さくシードし、パターン長を上限化。本番では走査対象のサイズ・深さに上限を設ける。
