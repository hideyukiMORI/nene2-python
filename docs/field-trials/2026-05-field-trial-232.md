## 概要

`fnmatch` は glob 形式（`*`, `?`, `[seq]`）のパターンマッチを提供する。HTTP API でラップし、名前マッチ・リストフィルタ・正規表現変換を検証した。ペンテスト回（232 % 4 = 0）として **ワイルドカード多用による DoS（ReDoS 的挙動）** と **`translate` 経由の正規表現インジェクション** を攻撃した。

| API | ユースケース |
|---|---|
| `fnmatch.fnmatchcase(name, pattern)` | OS 非依存（大小区別）の glob マッチ |
| `fnmatch.filter(names, pattern)` | 名前リストのフィルタ |
| `fnmatch.translate(pattern)` | glob → 正規表現変換 |

## 摩擦点

### F-1: `fnmatch.fnmatch` は OS により大小区別が変わる — `fnmatchcase` で決定化

**観察**: `fnmatch.fnmatch` は OS の `os.path.normcase` を通すため、**大文字小文字の区別が OS 依存**（macOS/Windows は区別しない、Linux は区別する）。同じコードが環境で異なる結果を出すと、認可判定などで事故になる。

**対処**: `fnmatch.fnmatchcase`（常に大小区別）を使い挙動を OS 非依存に固定。テストで `REPORT.TXT` が `*.txt` にマッチしないことを確認。

### F-2: ワイルドカード多用は性能劣化の懸念 — 数を上限化

**観察**: `*` や `?` を大量に含むパターンは、`translate` 後の正規表現で `.*` が増え、マッチ性能が劣化する可能性がある（理論上の DoS 懸念）。

**対処**: パターン長（200）に加え `* + ?` の総数を 20 に制限。ペンテストの実測では上限内（20 ワイルドカード × 500 文字名前）でも **3〜6ms** で完了し、現実的な ReDoS は観測されなかったが、入力制限を多層防御として維持。

### F-3: `translate` の出力は正規表現インジェクションされない

**観察**: `fnmatch.translate` はユーザーパターンを正規表現に変換するが、**glob メタ文字（`* ? [seq]`）以外の正規表現メタ文字はエスケープ**される。`(?P<x>.*)` を渡しても `(?s:\(.P<x>\..*\))\z` のように `(` `)` が `\(` `\)` にエスケープされ、名前付きグループ等の正規表現構文を注入できない。

**対処**: `translate` を信頼してよい（regex 注入耐性あり）。本 FT は変換結果を**表示するだけ**で `re.compile` して実行はしない。
