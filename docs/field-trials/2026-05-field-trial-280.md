## 概要

ReDoS（Regular expression Denial of Service）は、**壊滅的バックトラッキング**を起こすパターン（`(a+)+$` 等）に細工入力を与えて CPU を専有させる攻撃。Python の `re` はバックトラッキング型でネイティブ timeout がないため特に注意が要る。対策の核心は **① ユーザーに正規表現パターンを供給させない（固定許可リスト）、② パターンを線形に保つ、③ 入力長を制限**。ペンテスト回（280 % 4 = 0）として検証した。

| 対策 | 内容 |
|---|---|
| パターン固定 | 許可リストのみ（ユーザー regex 不可） |
| 線形パターン | ネスト量化子 `(a+)+` を避ける |
| 入力長制限 | re に timeout がないため必須 |

## 摩擦点

### F-1: ユーザーに正規表現パターンを供給させない

**観察**: ユーザー指定のパターンを `re.compile` すると、`(a+)+$` や `(.*a){20}` のような ReDoS パターンを注入され CPU 専有 DoS を受ける。`re` はバックトラッキング型でこれを防げない。

**対処**: パターンは `_PATTERNS` 許可リスト（名前指定）のみ。ユーザーは値だけを送り、パターンは選べない。診断で `(a+)+$` 等のパターン名指定がすべて 422。

### F-2: 採用パターンを線形に保つ

**観察**: 許可リストのパターン自体が ReDoS 脆弱だと意味がない。ネスト量化子（`(a+)+`）・選択肢の重複（`(a|a)*`）・隣接する貪欲量化子は危険。

**対処**: slug `^[a-z0-9]+(?:-[a-z0-9]+)*$` などは**各位置で曖昧さがなく線形**。ペンテストで 1000 文字の敵対的入力でも **3〜29ms** で完了（バックトラッキング爆発なし）。

### F-3: 入力長制限（re に timeout がない）

**観察**: Python の `re` はネイティブ timeout を持たない（`regex` サードパーティには timeout あり）。線形パターンでも入力が極端に長ければコストは増える。

**対処**: 入力長を 1000 に制限。`regex` モジュールの `timeout` 引数や `re` を別スレッド+タイムアウトで囲む手もあるが、本 FT は許可リスト + 線形 + 長さ制限で対処。
