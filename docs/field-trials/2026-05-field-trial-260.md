## 概要

zip 展開は二大脆弱性を抱える — **zip slip**（エントリ名の `../` や絶対パスで展開先ディレクトリ外に書き込み）と **zip bomb**（小さな圧縮データが巨大展開でディスク/メモリ枯渇）。HTTP API でラップし、エントリ名検証（`resolve` + `is_relative_to`）と展開サイズ上限で防御、ペンテストで攻撃した。FT225/FT226（解凍爆弾）・FT227（パストラバーサル）の統合的応用。

| 脅威 | 対策 |
|---|---|
| zip slip（`../`/絶対/`\`） | `resolve()` + `is_relative_to()` + バックスラッシュ拒否 |
| zip bomb | 宣言サイズ合計 + 実展開の両方を上限化 |
| エントリ数 DoS | エントリ数上限 |

## 摩擦点

### F-1: zip slip — `resolve()` + `is_relative_to()` で封じ込め

**観察**: エントリ名 `../../etc/passwd` や絶対パス `/tmp/evil` を展開すると**展開先の外**にファイルを書ける（zip slip / CVE 多数）。`zipfile.extract` は `..` を一部サニタイズするが、明示的に**拒否**する方が安全（FT227 と同思想）。

**対処**: `(base / name).resolve()` が `base.resolve()` 配下か `is_relative_to` で検証。`../../etc/passwd`・絶対パス・`a/../../b.txt` を 422。

### F-2【重要・発見】バックスラッシュの OS 依存 zip slip

**観察**: ペンテストで `..\..\evil.txt` が **Linux では 200**（`\` は Linux で通常のファイル名文字でありパス区切りでない → traversal にならない）。だが**同じアーカイブを Windows で展開すると `\` が区切りになり traversal**する。`pathlib` の判定はホスト OS のセマンティクスに従うため、Linux サーバーでは見逃す。

**対処**: エントリ名に `\` を含む場合は **OS 非依存に拒否**（実装中に発見し追加）。これで Windows へ配布されるアーカイブでも安全。

### F-3: zip bomb — 宣言サイズ + 実展開の二重上限

**観察**: 小さな zip（高圧縮ゼロ列）が展開後に巨大化する。`ZipInfo.file_size`（宣言サイズ）は嘘をつける可能性があるため、宣言値チェックだけでは不十分。

**対処**: ① 展開前に `sum(file_size)` で宣言サイズ合計を検査、② 実展開時も `read(MAX+1)` で打ち切る、の二重防御。10MB ゼロ列・10×1MB をいずれも 30ms で 422。エントリ数も上限化。
