## 概要

`uuid` は各種バージョンの UUID を生成する。HTTP API でラップし生成・解析を検証した。セキュリティ上の核心は **uuid1（時刻+MAC ベース）と uuid4（乱数ベース）の違い** — uuid1 はノード（MAC アドレス）とタイムスタンプを内包し**予測・追跡可能**なため、推測困難性が必要なトークン・識別子には uuid4（または `secrets`、FT203）を使う。

| API | 性質 | 用途 |
|---|---|---|
| `uuid.uuid4()` | 乱数（122bit） | 識別子・推測されたくない ID |
| `uuid.uuid1()` | 時刻+MAC | **予測可能**。トークン非推奨 |
| `uuid.uuid5(ns, name)` | 名前ベース（SHA-1・決定的） | 同じ名前→同じ UUID |

## 摩擦点

### F-1: uuid1 は予測可能・MAC 漏洩 — トークンには uuid4 を使う

**観察**: `uuid.uuid1()` はタイムスタンプとノード識別子（多くの環境で MAC アドレス）から生成され、**生成時刻とホストを推測**できる。連番性もあるため、これをセッショントークンや推測されたくない ID に使うと列挙攻撃・情報漏洩につながる。

**対処**: 識別子・トークンは `uuid4()`（122bit の乱数）を既定にする。診断で uuid4 を 1000 個生成し全てユニーク・version=4 を確認。さらに高保証が要るトークンは `secrets.token_urlsafe`（FT203）を使う。

### F-2: 受け取った UUID の version を検証して用途を制限する

**観察**: 外部から受け取った UUID をそのまま信頼すると、攻撃者が uuid1 等を渡して時刻ベースの推測を狙う余地がある。

**対処**: `parse_uuid` で `version` を返し `is_random_v4` フラグで「乱数ベースか」を明示。トークン検証では v4 のみ受理するポリシーを組める。診断で v1 が `is_random_v4=False` になることを確認。

### F-3: `uuid.UUID()` は寛容パーサ（波括弧・urn・ハイフン無し）

**観察**: `uuid.UUID("{...}")` や `urn:uuid:...`、ハイフン無し 32 桁も受理する。厳格な形式一致を期待すると意外。一方、不正文字列は `ValueError`。

**対処**: 寛容性を理解しつつ、不正は 422 に変換。`'; DROP TABLE--` 等の注入文字列は ValueError で 422（SQL 等に渡る前に弾ける）。
