## 概要

`graphlib.TopologicalSorter` は依存グラフをトポロジカルソート（依存先が先に来る順序）し、循環依存を `CycleError` で検出する。HTTP API でラップし依存解決と循環検出を検証した。ビルド順序・タスク依存・マイグレーション順序の決定に有用。

| API | ユースケース |
|---|---|
| `TopologicalSorter.add(node, *deps)` | ノードと依存を登録 |
| `.static_order()` | 依存順のリスト（循環なら CycleError） |

## 摩擦点

### F-1: 循環依存は `CycleError`、循環ノードは `args[1]` に入る

**観察**: 循環依存（`a→b→a`）があると `static_order()` は `CycleError` を送出する。エラーメッセージだけでなく `error.args[1]` に**循環に含まれるノード列**が入っており、デバッグに有用。

**対処**: `CycleError` を捕捉し `args[1]` から循環パスを抽出して 422 メッセージに含める。自己循環（`a→a`）も検出。

### F-2: `add(node, *deps)` の方向（依存先が先）

**観察**: `add(node, *predecessors)` は「node は predecessors の**後**に来る」を意味する。`static_order()` は依存先（predecessor）が先に並ぶ。方向を取り違えると逆順になる。

**対処**: グラフを「node → 依存先リスト」で受け取り、`add(node, *deps)` で登録。`base→lib→app` の順（依存先が先）を確認。

### F-3: ノード/エッジ数の上限（DoS）

**観察**: 巨大なグラフはソートのコストが増す。`graphlib` 自体に上限はない。

**対処**: ノード数（1000）・ノードあたり依存数（100）・名前長を上限化。
